## Section11

### 원자적 연산
#### 원자적 연산
* 원자적 연산은 멀티스레드 상황에서 아무런 문제가 발생하지 않는다. 
* 하지만 원자적 연산이 아닌 경우에는 synchronized 블럭이나 Lock 등을 사용해서 안전한 임계 영역을 만들어야 한다.

#### AtomicInteger
* 자바에서 제공하는 멀티스레드 상황에서 안전하게 증가 연산을 수행할 수 있는 클래스
* CAS 연산을 사용한다.

#### 성능 비교
* BasicInteger
  * 가장 빠르다. 
  * CPU 캐시를 적극 사용한다. CPU 캐시의 위력을 알 수 있다. 
  * 안전한 임계 영역도 없고, volatile 도 사용하지 않기 때문에 멀티스레드 상황에는 사용할 수 없다. 
  * 단일 스레드가 사용하는 경우에 효율적이다.
* VolatileInteger 
  * volatile 을 사용해서 CPU 캐시를 사용하지 않고 메인 메모리를 사용한다. 
  * 안전한 임계 영역이 없기 때문에 멀티스레드 상황에는 사용할 수 없다. 
  * 단일 스레드가 사용하기에는 BasicInteger 보다 느리다. 그리고 멀티스레드 상황에도 안전하지 않다.
* SyncInteger
  * synchronized 를 사용한 안전한 임계 영역이 있기 때문에 멀티스레드 상황에도 안전하게 사용할 수 있다.
  * MyAtomicInteger 보다 성능이 느리다.
* MyAtomicInteger 
  * 자바가 제공하는 AtomicInteger 를 사용한다. 멀티스레드 상황에 안전하게 사용할 수 있다. 
  * 성능도 synchronized , Lock(ReentrantLock) 을 사용하는 경우보다 1.5 ~ 2배 정도 빠르다
  *  AtomicInteger 가 제공하는 incrementAndGet() 메서드는 락을 사용하지 않고, 원자적 연산을 만들어낸다. (그래서 빠른거다)

### CAS 연산
* 락을 걸지 않고 원자적인 연산을 수행할 수 있는 방법
* 다만 락을 완전히 대체하는 것은 아니고, 작은 단위의 일부 영역에 적용할 수 있다.
* 원자적이지 않은 두 개의 연산을 CPU 하드웨어 차원에서 특별하게 하나의 원자적인 연산으로 묶어서 제공하는 기능 (하드웨어가 제공하는 기능)
* (직접 쓸일이 거의 없긴하다.)

#### Lock VS CAS
* Lock
  * 비관적 접근 방식(충돌이 발생할 것으로 간주)
  * 데이터 접근 전에 항상 락을 획득
  * 다른 스레드 접근을 막음
* CAS
  * 낙관적 접근 방식(충돌이 발생하지 않을 것으로 간주)
  * 락을 사용하지 않고 데이터에 바로 접근
  * 충돌이 발생한다면 그냥 다시 시도

#### CAS 언제 써야 하나
* 간단한 CPU 연산의 경우 충돌이 거의 발생하지 않는다.
* CAS는 락을 얻지 않기에 훨씬 속도가 빠르다.
* 따라서 간단한 CPU 연산의 경우 CAS 방식이 더 적합하다.

### CAS 락 구현
#### 스핀락
* CAS 연산을 기반으로 락을 구현할 수 있다.
  * CAS 연산을 기반으로 반복문을 돌며 대기(spin-wait or busy-wait)하는 방식으로 구현
  * SpinLock.java 참고
* 해당 방식의 경우 스레드가 상태를 변경하지 않고 대기하게 된다.
  * 컨텍스트 스위칭 등의 비용이 무시되기에 매우 가볍다.
* 하지만 상태를 변경하지 않는다는 것은 runnable 상태를 유지하여 대기한다는 것이다.
  * 즉, 대기하는 와중에도 스레드는 CPU를 사용하고 있다.
* 만약 오래 걸리는 연산에서 스핀락 사용하게 되면 CPU를 계속 사용하며 기다리는 최악의 경우가 발생할 수 있다.
* 따라서 스핀락은 금방 끝나며 충돌 가능성이 연산에서 사용해야 한다.
  * 스핀락 대상 연산
    * 숫자 값의 증가
    * 자료 구조의 데이터 추가 및 삭제
